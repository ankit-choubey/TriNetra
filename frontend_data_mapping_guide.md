# Trinetra Agent Integration: Frontend Data Mapping Guide

The Trinetra AI Agents output a highly refined, deterministic intelligence payload into the Unified Credit Scoring Object (UCSO). 

The frontend needs to be updated to map its UI components to the **actual data structure** produced by the AI Agent pipeline, rather than expecting arbitrary or mocked keys.

This guide outlines exactly how the frontend should pull data from the `ucsoData` object.

---

## 1. Risk Ring & Decision Dashboards

The `RecommendationEngine.jsx` and `ResearchAgent.jsx` pages rely heavily on risk scoring. The Risk Agent outputs into the `risk` namespace.

**Current Frontend Expectation:**
```javascript
// What frontend currently looks for:
const confidence = ucsoData.risk.agent_confidence; // 0-1
const grade = ucsoData.risk.risk_grade; // "A+", "B"
const prob = ucsoData.risk.default_probability; // "low", "medium"
```

**Required Frontend Update:**
```javascript
// What the Agent ACTUALLY sends:
// The score is a pure risk score (0.0 to 1.0) where 1.0 is MAXIMUM risk and 0.0 is MINIMUM risk.
const riskScore = ucsoData.risk.score; 

// If you want a "Confidence" percentage (higher is safer) for the UI Ring:
const confidencePercentage = (1.0 - riskScore) * 100;

// The Band represents the default probability category. 
// Possible values: "LOW", "MEDIUM", "HIGH", "REJECT"
const band = ucsoData.risk.band; 

// The final actionable decision.
// Possible values: "APPROVE", "REVIEW", "REJECT"
const decision = ucsoData.risk.decision;
```

## 2. SHAP Values (Explainability Chart)

The frontend's SHAP values component expects a flat key-value map. The Risk Agent actually provides a richer array format that includes the human-readable feature name, its weight, and its SHAP contribution.

**Current Frontend Expectation:**
```javascript
// Flat dictionary
const shapData = ucsoData.risk.shap_values; 
// e.g. { "dscr": 0.15, "leverage": 0.05 ... }
```

**Required Frontend Update:**
```javascript
// The Agent provides a dedicated top_risk_factors sorted array:
const topFactors = ucsoData.risk.top_risk_factors;

/*
Structure received from Agent:
[
  {
    "feature": "dscr_normalized",
    "shap_value": 0.1423,
    "weight": 0.25
  },
  {
    "feature": "leverage_normalized",
    "shap_value": 0.0891,
    "weight": 0.18
  }
]
*/

// Suggested frontend mapping for Recharts:
const chartData = topFactors.map(factor => ({
    name: factor.feature.replace('_normalized', '').replace('_norm', '').toUpperCase(),
    contribution: (factor.shap_value || factor.contribution) * 100
}));
```

## 3. Rejection Reasons & Corrective Actions

The frontend should surface the actionable insights generated by the model. The Agent provides these in the `risk` namespace.

**Required Frontend Update:**
Add UI components to display these string arrays in the Recommendation Engine:
```javascript
// Array of string explanations why the loan was flagged
const reasons = ucsoData.risk.rejection_reasons; 
// e.g. ["Debt Service Coverage Ratio is below acceptable threshold"]

// Array of string suggestions for the applicant
const corrective = ucsoData.risk.corrective_actions;
// e.g. ["Improve DSCR by reducing debt or increasing cash flows"]
```

## 4. Assessment Breakdown Chart (The 5 Bars)

The frontend's Bar Chart looks for arbitrary scores (`cash_flow_score`, `management_score`, etc.) in `derived_features` or `financials` that our pure ML pipeline does not artificially invent. 

We generate **pure quantifiable metrics**.

**Current Frontend Expectation:**
Looks for fake 0-100 scores in `derived_features.cash_flow_score`, etc.

**Required Frontend Update:**
Map the Bar Chart to the **actual normalized features** calculated by the Risk Agent before the final decision.

Pull from: `ucsoData.risk.feature_vector`

*   **Capacity Metric:** Use `(1.0 - feature_vector.dscr_normalized) * 100`
*   **Leverage Metric:** Use `(1.0 - feature_vector.leverage_normalized) * 100`
*   **Operating Metric:** Use `(1.0 - feature_vector.ebitda_margin_normalized) * 100`
*   **Compliance Metric:** Use `(1.0 - feature_vector.gst_discrepancy_norm) * 100`
*   **Management/Character Metric:** Use `(1.0 - feature_vector.news_sentiment_norm) * 100`

*(Note: We inverse them `1.0 - X` because the agent outputs risk (higher = worse), but charts usually look better when higher = safer).*

## 5. Intelli-Credit Score

The frontend looks for a random 0-900 score in `derived_features.credit_score`. 

**Required Frontend Update:**
CIBIL score is extracted directly from documents by the Doc Intelligence agent. 
If the UI wants to show an internal "Intelli-Credit" score on a 900 scale, derive it mathematically from the agent's risk score in the frontend:

```javascript
// Calculate a 300 to 900 scale score based on the Agent's risk score
const agentRisk = ucsoData.risk.score; // 0.0 to 1.0
const intelliScore = Math.round(900 - (agentRisk * 600)); 
```

## 6. Limits & Rates

Ensure formatting is applied to the exact keys output by the Risk Agent:
```javascript
const maxLimit = ucsoData.risk.recommended_limit; // Float (e.g. 45000000.0)
const intRateBps = ucsoData.risk.recommended_rate_bps; // Float (e.g. 963.8)
const intRatePct = intRateBps / 100.0; // Display as 9.64%
```

## 7. Compliance Checklist

The frontend expects a single `compliance.compliance_score`. The Compliance Agent is a strict boolean gatekeeper.

**Required Frontend Update:**
Map the UI strictly to the missing documents array provided by the Agent:
```javascript
const complianceStatus = ucsoData.compliance.status; // "PASS" or "FAIL"
const missingDocs = ucsoData.compliance.missing_documents; // Array of expected doc_types

// If calculating a percentage for a progress bar:
const requiredTotal = 4;
const missingCount = missingDocs.length;
const complianceProgressPct = ((requiredTotal - missingCount) / requiredTotal) * 100;
```
